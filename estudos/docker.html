Fundamentos

O que é Docker?

Entenda os conceitos fundamentais da containerização e por que ela revolucionou o desenvolvimento de software.

Containers

Containers são pacotes leves e portáteis que incluem tudo o que uma aplicação precisa para rodar: código, runtime, ferramentas, bibliotecas e configurações. Diferente de VMs, containers compartilham o kernel do SO, tornando-os extremamente eficientes.

Imagens

Imagens Docker são templates imutáveis e read-only que servem como base para criar containers. Cada imagem é construída em camadas (layers), permitindo reutilização eficiente e versionamento. São armazenadas em registries como Docker Hub.

Networks

Redes Docker permitem que containers se comuniquem entre si e com o mundo exterior. O driver bridge é o padrão para comunicação local, overlay para clusters multi-host, host para máxima performance e none para isolamento total.

Volumes

Volumes são o mecanismo preferido para persistir dados gerados por containers. Gerenciados pelo Docker, eles sobrevivem à recriação de containers e podem ser compartilhados entre múltiplos containers, garantindo integridade dos dados.

Dockerfile

Um arquivo de texto com instruções para construir uma imagem Docker automatizada. Define a imagem base, dependências, variáveis de ambiente, portas expostas e comandos de inicialização. Suporta multi-stage builds para otimização.

Docker Compose

Ferramenta para definir e executar aplicações multi-container. Utiliza um arquivo YAML para configurar serviços, redes e volumes. Com um único comando (docker-compose up), toda a stack é provisionada.

Figura 1: Arquitetura Docker - Containers compartilhando volumes e redes

Dica Profissional: Containers não são VMs! Eles compartilham o kernel do host, tornando-se mais leves (MB vs GB) e com startup em segundos. Use containers para aplicações e VMs quando precisar de isolamento completo de kernel.

Docker em Slides

O que é Containerização?

Containerização é a tecnologia de empacotar uma aplicação e todas as suas dependências em um único pacote padronizado. Isso garante que a aplicação rode de forma idêntica em qualquer ambiente: desenvolvimento, staging ou produção.

  • Isolamento de processos
  • Portabilidade entre ambientes
  • Eficiência de recursos
  • Escalabilidade horizontal
2 / 8

Camadas (Layers)

Imagens Docker são construídas em camadas sobrepostas. Cada instrução no Dockerfile cria uma nova camada. Camadas são cacheadas e reutilizadas, tornando builds rápidos e eficientes em disco.

  • Base Layer: Sistema operacional mínimo
  • Dependency Layer: Bibliotecas e pacotes
  • Application Layer: Código fonte
  • Container Layer: Camada writable em runtime
3 / 8

Tipos de Rede

Docker oferece múltiplos drivers de rede para diferentes cenários de comunicação entre containers e com o host.

  • Bridge: Padrão. Comunicação isolada entre containers
  • Host: Compartilha stack de rede do host
  • Overlay: Comunicação multi-host (Swarm)
  • Macvlan: Atribui MAC address real ao container
  • None: Isolamento total de rede
4 / 8

Persistência de Dados

Por padrão, dados em containers são efêmeros. Docker oferece três métodos para persistência:

  • Volumes: Gerenciados pelo Docker, melhor opção para produção
  • Bind Mounts: Mapeiam diretórios específicos do host
  • tmpfs: Armazenamento em memória RAM (temporário)
5 / 8

Segurança

Segurança em Docker é multicamadas e deve ser considerada desde o desenvolvimento até a produção.

  • Execute containers como non-root user
  • Use imagens oficiais e escaneie vulnerabilidades
  • Implemente multi-stage builds para reduzir superfície de ataque
  • Configure limites de recursos (CPU/Memória)
  • Isole redes com policies e firewalls
6 / 8

Docker Compose

Orchestrando aplicações multi-serviço com um único arquivo YAML declarativo.

  • Define serviços, redes e volumes em um só lugar
  • Gerencia dependências entre serviços
  • Facilita replicação de ambientes
  • Suporta overrides para diferentes ambientes
  • Integração nativa com Docker Swarm
7 / 8

Monitoramento

Containers precisam de observabilidade. Ferramentas essenciais:

  • docker stats: Métricas em tempo real
  • docker logs: Coleta de logs centralizada
  • Prometheus + Grafana: Métricas e dashboards
  • cAdvisor: Análise de uso de recursos
  • Health Checks: Verificação de saúde nativa
8 / 8

Produção & Cloud

Deploy de containers em ambientes cloud-native:

  • AWS: ECS, EKS, Fargate, App Runner
  • Azure: ACI, AKS, Container Apps
  • GCP: Cloud Run, GKE
  • On-Premise: Docker Swarm, Kubernetes
  • CI/CD: GitHub Actions, GitLab CI, Jenkins

Comandos Essenciais

Guia de referência rápida com os comandos mais utilizados no dia a dia profissional.

Gerenciamento de Containers

Containers - Comandos Básicos BASH 
# Criar e executar um container
docker run -d -p 80:80 --name meu-nginx nginx

# Listar containers em execução
docker ps

# Listar todos os containers (incluindo parados)
docker ps -a

# Parar um container
docker stop meu-nginx

# Iniciar um container parado
docker start meu-nginx

# Reiniciar um container
docker restart meu-nginx

# Remover um container (deve estar parado)
docker rm meu-nginx

# Forçar remoção (running)
docker rm -f meu-nginx

# Executar comando dentro do container
docker exec -it meu-nginx bash

# Ver logs em tempo real
docker logs -f meu-nginx

# Estatísticas de uso de recursos
docker stats

Gerenciamento de Imagens

Images - Build & Registry BASH 
# Construir imagem a partir do Dockerfile
docker build -t minha-app:v1 .

# Construir sem cache (build limpo)
docker build --no-cache -t minha-app:v1 .

# Listar imagens locais
docker images

# Remover uma imagem
docker rmi minha-app:v1

# Remover imagens dangling (sem tag)
docker image prune

# Taggear imagem para push
docker tag minha-app:v1 usuario/minha-app:latest

# Enviar para Docker Hub
docker push usuario/minha-app:latest

# Baixar imagem do registry
docker pull nginx:alpine

# Inspecionar metadados da imagem
docker inspect nginx:alpine

# Histórico de camadas da imagem
docker history minha-app:v1

# Salvar imagem como arquivo tar
docker save -o minha-app.tar minha-app:v1

# Carregar imagem de arquivo tar
docker load -i minha-app.tar

Volumes & Networks

Volumes e Networks BASH 
# Criar um volume
docker volume create meu-volume

# Listar volumes
docker volume ls

# Inspecionar volume
docker volume inspect meu-volume

# Remover volume não utilizado
docker volume rm meu-volume

# Remover volumes não utilizados
docker volume prune

# Criar rede bridge customizada
docker network create --driver bridge minha-rede

# Listar redes
docker network ls

# Conectar container à rede
docker network connect minha-rede meu-container

# Desconectar container da rede
docker network disconnect minha-rede meu-container

# Inspecionar rede
docker network inspect minha-rede

Docker Compose

Docker Compose - Comandos BASH 
# Iniciar todos os serviços em background
docker-compose up -d

# Parar e remover containers, networks e volumes
docker-compose down

# Parar sem remover volumes
docker-compose down -v

# Reconstruir imagens antes de subir
docker-compose up -d --build

# Escalar um serviço para 3 réplicas
docker-compose up -d --scale web=3

# Ver logs de todos os serviços
docker-compose logs -f

# Ver logs de serviço específico
docker-compose logs -f web

# Executar comando em serviço
docker-compose exec web bash

# Listar containers do compose
docker-compose ps

# Validar arquivo compose
docker-compose config

Dockerfile Profissional

Exemplos de Dockerfiles otimizados para diferentes cenários de produção.

Aplicação Node.js (Multi-Stage)

Dockerfile - Node.js Production DOCKERFILE 
# Estágio 1: Build
FROM node:18-alpine AS builder
WORKDIR /app

# Copiar dependências primeiro (cache otimizado)
COPY package*.json ./
RUN npm ci --only=production

# Copiar código fonte e buildar
COPY . .
RUN npm run build

# Estágio 2: Produção (imagem mínima)
FROM node:18-alpine
WORKDIR /app

# Criar usuário não-root para segurança
RUN addgroup -g 1001 -S nodejs && adduser -S nodejs -u 1001

# Copiar apenas artefatos do build
COPY --from=builder --chown=nodejs:nodejs /app/dist ./dist
COPY --from=builder --chown=nodejs:nodejs /app/node_modules ./node_modules
COPY --from=builder --chown=nodejs:nodejs /app/package.json ./

USER nodejs

EXPOSE 3000

HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD node healthcheck.js || exit 1

CMD ["node", "dist/main.js"]

Aplicação Python (Django/Flask)

Dockerfile - Python DOCKERFILE 
FROM python:3.11-slim

WORKDIR /app

# Variáveis de ambiente
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PIP_NO_CACHE_DIR=1 \
    PIP_DISABLE_PIP_VERSION_CHECK=1

# Instalar dependências do sistema
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential libpq-dev && \
    rm -rf /var/lib/apt/lists/*

# Instalar dependências Python
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copiar aplicação
COPY . .

# Usuário não-root
RUN useradd -m -u 1000 appuser && chown -R appuser:appuser /app
USER appuser

EXPOSE 8000

CMD ["gunicorn", "--bind", "0.0.0.0:8000", "--workers", "4", "app:application"]

Docker Compose - Stack Completa

docker-compose.yml - Full Stack YAML 
version: '3.8'

services:
  web:
    build:
      context: ./web
      dockerfile: Dockerfile.prod
    container_name: web-app
    restart: unless-stopped
    ports:
      - "80:80"
    environment:
      - NODE_ENV=production
      - DATABASE_URL=postgres://db:5432/app
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started
    networks:
      - frontend
      - backend
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M

  db:
    image: postgres:15-alpine
    container_name: postgres-db
    restart: unless-stopped
    environment:
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: app
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d app"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - backend

  redis:
    image: redis:7-alpine
    container_name: redis-cache
    restart: unless-stopped
    volumes:
      - redis_data:/data
    command: redis-server --appendonly yes --maxmemory 256mb
    networks:
      - backend

  nginx:
    image: nginx:alpine
    container_name: reverse-proxy
    restart: unless-stopped
    ports:
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
      - ./ssl:/etc/nginx/ssl:ro
    depends_on:
      - web
    networks:
      - frontend

volumes:
  postgres_data:
    driver: local
  redis_data:
    driver: local

networks:
  frontend:
    driver: bridge
  backend:
    driver: bridge
    internal: true
FAQ

Perguntas & Respostas

As dúvidas mais comuns sobre Docker, respondidas de forma clara e objetiva.

1. Qual a diferença entre Docker e Máquina Virtual (VM)?

VMs virtualizam o hardware inteiro, incluindo um sistema operacional completo (hipervisor + Guest OS), consumindo GBs de RAM e minutos para iniciar. Docker utiliza containerização no nível do sistema operacional, compartilhando o kernel do host. Containers são medidos em MB, iniciam em segundos e oferecem densidade muito maior (você pode rodar centenas de containers onde cabem poucas VMs). VMs são ideais quando você precisa de isolamento completo de kernel ou rodar diferentes SOs.

2. O que é uma camada (layer) em uma imagem Docker?

Cada instrução em um Dockerfile (RUN, COPY, ADD) cria uma nova camada read-only sobre a imagem base. Camadas são cacheadas e reutilizadas entre builds. Quando um container é executado, uma camada writable é adicionada no topo. Se dois containers usam a mesma imagem base, eles compartilham as camadas subjacentes, economizando espaço em disco. A ordem das instruções importa: coloque comandos que mudam com menos frequência no topo do Dockerfile para maximizar o cache.

3. Como persistir dados quando um container é removido?

Use Volumes (recomendado para produção), gerenciados pelo Docker em /var/lib/docker/volumes/, ou Bind Mounts para mapear diretórios específicos do host. Volumes são independentes do ciclo de vida do container, podem ser nomeados, compartilhados entre múltiplos containers e facilmente backupados. Para dados temporários sensíveis, use tmpfs mounts (armazenamento em RAM). Nunca armazene dados importantes na camada writable do container.

4. Como containers se comunicam entre si?

Por padrão, containers na mesma rede bridge customizada podem se comunicar usando os nomes dos containers como hostname (DNS interno do Docker). No Docker Compose, serviços resolvem automaticamente pelo nome do serviço. Para comunicação entre hosts diferentes, use redes overlay (Docker Swarm) ou orquestradores como Kubernetes. A porta exposta no container (EXPOSE) deve ser mapeada para o host (-p host:container) apenas quando necessário acesso externo.

5. O que é Docker Compose e quando usar?

Docker Compose é uma ferramenta para definir e gerenciar aplicações multi-container através de um arquivo YAML declarativo. Use quando sua aplicação depende de múltiplos serviços (ex: web + API + banco + cache). Com um único docker-compose up, toda a stack é provisionada com redes e volumes configurados automaticamente. É ideal para desenvolvimento local, testes de integração e ambientes de staging. Para produção em grande escala, considere Docker Swarm ou Kubernetes.

6. Como otimizar o tamanho das imagens Docker?

Use Multi-stage builds para separar ambiente de build e produção. Prefira imagens base Alpine ou Distroless ao invés de Debian/Ubuntu full. Combine comandos RUN em uma única camada para reduzir layers. Remova caches de pacotes (rm -rf /var/cache/apk/*). Use .dockerignore para excluir arquivos desnecessários do contexto de build. Ordene instruções do menos mutável ao mais mutável para maximizar cache.

7. É seguro rodar containers como root?

Não. Sempre crie e use um usuário não-root no Dockerfile (USER appuser). Containers como root que sofrem escape de container comprometem o host inteiro. Configure --security-opt=no-new-privileges, limite capabilities com --cap-drop ALL, use read-only filesystem (--read-only) quando possível e defina limites de recursos. Ferramentas como Docker Scout e Trivy ajudam a escanear vulnerabilidades em imagens.

8. Como fazer backup de volumes Docker?

Use um container temporário para criar tarballs dos volumes: docker run --rm -v meu-volume:/data -v $(pwd):/backup alpine tar czf /backup/backup.tar.gz -C /data . Para restore: docker run --rm -v meu-volume:/data -v $(pwd):/backup alpine sh -c "cd /data && tar xzf /backup/backup.tar.gz". Para automação, considere plugins como docker-volume-backup ou integre com ferramentas de backup corporativas.

9. O que fazer quando o Docker consome muito disco?

Execute docker system df para diagnosticar. Limpe com docker system prune -a (remove tudo não utilizado: containers parados, imagens dangling, networks não usadas, cache de build). Para limpeza seletiva: docker volume prune, docker image prune, docker container prune. Configure políticas de retenção de logs no daemon.json. Monitore regularmente o uso de disco em produção para evitar interrupções.

10. Docker Swarm vs Kubernetes: qual escolher?

Docker Swarm é mais simples, nativo no Docker, ideal para times pequenos e clusters menores (até ~50 nós). Kubernetes é o padrão de fato da indústria, oferece ecossistema vasto, auto-scaling avançado, service mesh, operators e suporte multi-cloud. Para aprendizado e pequenos projetos, comece com Swarm. Para produção enterprise, Kubernetes é indispensável. Certificações: DCA (Docker), CKA/CKAD (Kubernetes).

11. Como implementar Health Checks?

No Dockerfile: HEALTHCHECK --interval=30s --timeout=3s --retries=3 CMD curl -f http://localhost:8080/health || exit 1. No docker-compose: use a seção healthcheck com test, interval, timeout, retries. O Docker monitora o status (starting, healthy, unhealthy). Orquestradores usam essa informação para restart automático e load balancing. Health checks devem ser leves e verificar dependências críticas (banco, cache) sem sobrecarregar.

12. Como passar variáveis sensíveis (secrets) para containers?

Nunca commite secrets em imagens! Use: (1) Variáveis de ambiente em runtime (-e ou arquivo .env no host, não no container). (2) Docker Secrets (Swarm mode): echo "senha" | docker secret create db_password -. (3) Vaults externos: HashiCorp Vault, AWS Secrets Manager, Azure Key Vault. (4) tmpfs mounts para montar secrets em memória RAM. (5) Ferramentas de runtime como docker-credential-helpers.

13. O que é Docker Hub e registries privados?

Docker Hub é o registry público padrão da Docker, com milhões de imagens oficiais e da comunidade. Para imagens proprietárias, use registries privados: Docker Hub Private, AWS ECR, Azure ACR, GCP GCR, GitHub Container Registry, Harbor (open-source self-hosted), ou docker registry básico. Registries privados oferecem controle de acesso, scanning de vulnerabilidades, retenção de imagens e compliance.

14. Como escalar containers horizontalmente?

Com Docker Compose: docker-compose up -d --scale web=5. Com Docker Swarm: docker service scale meu-servico=10. Com Kubernetes: kubectl scale deployment meu-app --replicas=10. O load balancer distribui tráfego entre réplicas. Para auto-scaling automático baseado em métricas (CPU, memória, requisições), use Kubernetes HPA (Horizontal Pod Autoscaler) ou soluções cloud como AWS ECS Service Auto Scaling.

15. Como debugar um container que não inicia?

Passos sistemáticos: (1) docker logs container_id - verifique erros de aplicação. (2) docker inspect container_id - examine config, mounts, env vars, estado. (3) docker events - acompanhe eventos em tempo real. (4) Execute com --entrypoint sh para inspecionar manualmente. (5) Verifique portas em conflito (netstat). (6) Teste localmente fora do Docker. (7) Use docker exec -it container_id sh se o container estiver rodando mas com problemas.
